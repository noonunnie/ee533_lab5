# Lab5 - Pipelined Processor Integration with NetFPGA

## Overview

This project integrates a **5-stage pipelined processor** (from your PipelinedDatapath.v schematic) into the NetFPGA reference router architecture. The processor is completely **isolated from packet routing** and communicates exclusively through the **register interface**, following the same pattern as the Mini-IDS lab.

### Key Features

1. **Processor Isolation**: Processor is completely separated from packet routing
   - No packet data enters the processor
   - All communication through dedicated register interface
   - Operates independently on private instruction and data memory

2. **Register-Based Control**: Following the Mini-IDS pattern
   - Write instructions via `proc_inst_write` register
   - Read program counter via `proc_pc` register
   - Read register file via `proc_reg0`, `proc_reg1`, `proc_reg2`, `proc_reg3` registers
   - Send commands via `proc_cmd` register (reset, enable)

3. **Register Ring Architecture**
   - Processor added to the register ring after output queues
   - Register requests flow: Core → Router Stages → Processor → Core
   - Non-blocking: Processor responds to register requests independently

## Register Interface

### The 7 Processor Registers

All registers are 32-bit. Register addresses will be generated by the build system.

| Register | Generated Address | R/W | Purpose |
|----------|-------------------|-----|---------|
| **proc_inst_write** | 0x2000200 (example) | W | Write instruction to instruction memory |
| **proc_pc** | 0x2000204 (example) | R | Read program counter (current instruction address) |
| **proc_cmd** | 0x2000208 (example) | W | Command: [0]=reset, [1]=enable |
| **proc_reg0** | 0x200020C (example) | R | Read Register File[0] (lower 32 bits) |
| **proc_reg1** | 0x2000210 (example) | R | Read Register File[1] (lower 32 bits) |
| **proc_reg2** | 0x2000214 (example) | R | Read Register File[2] (lower 32 bits) |
| **proc_reg3** | 0x2000218 (example) | R | Read Register File[3] (lower 32 bits) |

**Note**: Actual addresses generated by `nf_register_gen.pl` from `processor.xml`

### Software Usage Example

```bash
# Write instruction 0x60000000 to instruction memory
regwrite 0x2000200 0x60000000

# Reset processor
regwrite 0x2000208 0x00000001

# Enable processor 
regwrite 0x2000208 0x00000003   # bit[0]=1 (reset), bit[1]=1 (enable)

# Read program counter
regread 0x2000204

# Read processor register 2
regread 0x2000214
```

## Processor Control Flow

### Writing an Instruction

```
Software Command:
regwrite(PROC_INST_WRITE_REG, 0x60000000)
         ↓
Register Request on ring
         ↓
processor_wrapper receives request
         ↓
inst_wr_en signal asserted
         ↓
PipelinedDatapath.InstructionMem
         ↓
Instruction stored at memory location
```

### Reading Processor State

```
Software Command:
value = regread(PROC_PC_REG)
         ↓
Register Request on ring
         ↓
processor_wrapper receives request
         ↓
Combinational logic selects PC output
         ↓
reg_data_out = current PC value
         ↓
Linux application receives current instruction address
```

## Command Register Bits

**proc_cmd register** (0x2000208):

```
Bit 0: RESET
       1 = Reset processor and clear all state
       0 = Normal operation

Bit 1: ENABLE
       1 = Enable instruction pipeline
       0 = Hold processor (for debugging)

Bits [31:2]: Reserved for future use
```

**Typical sequence:**

```
1. regwrite(PROC_CMD_REG, 0x00000001)  # Reset = 1, Enable = 0
2. regwrite(PROC_CMD_REG, 0x00000003)  # Reset = 1, Enable = 1 (release reset)
3. regwrite(PROC_CMD_REG, 0x00000002)  # Reset = 0, Enable = 1 (normal run)
```

## Data Flow: Instruction Example

### Load Address 0 into Reg2

**Instruction**: `0x60000000`
```
┌─ Bit Field ────┬─ Value ─┬─ Meaning ────────────┐
│ [31:26]        │  6'b011 │ Opcode: Load         │
│ [25:20]        │  000000 │ Address = 0          │
│ [19:17]        │  010    │ Destination = Reg2   │
└────────────────┴─────────┴──────────────────────┘
```

**Execution flow:**

```
Cycle 1 (IF):  PC = 0, Fetch instruction 0x60000000
Cycle 2 (ID):  Decode: Load from addr 0 into Reg2
Cycle 3 (EX):  Calculate address = 0
Cycle 4 (MEM): Read data memory[0] = 0x00000004
Cycle 5 (WB):  Write Reg2 = 0x00000004

After 5 cycles:
regread(PROC_REG2_REG) returns 0x00000004
```

### Common Issues

| Issue | Cause | Solution |
|-------|-------|----------|
| Register access fails | Wrong addresses | Verify `reg_defines_lab5.h` |
| PC doesn't advance | Processor not enabled | Write 0x3 to cmd register |
| All registers return 0 | Processor not in register ring | Check `processor_wrapper` connections |
| Instruction not write | Wrong memory address | Check `PROC_INST_WRITE_REG` |

## Register Interface Ring

The processor connects into the **register ring architecture** after the output queues:

```
reg_req/ack/addr/data flow:
Core → Arbiter → OPLookup → IDS → OutputQueues → Processor → Master → Core
```

This ensures:
- ✅ All modules see every register request
- ✅ Each module can respond to its address range
- ✅ Processor isolated from data path
- ✅ Processor commands from software work identically to IDS
